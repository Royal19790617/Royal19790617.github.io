package.json与package.lock.json

先说package.json是干什么的

  • 又要拿java来做比较，这就相当于springboot中的pom.xml文件，主要是用来管理依赖包的，包含项目名称，版本号，依赖名称，对应版本号等等
认识下package.json中的一些关键节点含义
name 项目名称
version 项目的版本号
scripts 启动脚本入口

  • 通过npm命令启动项目，譬如npm run start启动项目，npm run test启动测试用例

    dependencies 依赖包及版本号

  • 运行初始化命令后，会把koa相关依赖包导入项目，放置于项目根目录下的node_modules目录中

  • 安装命令如下

细心的同学肯定发现了，在版本号区域有两种符号 ^ 和 ~ ，这个是啥意思呢？
  • 版本号的第一位是大版本号，会锁定，不会改变
  • ^ 代表的是三位版本号中的第二位向上兼容，当前项目中koa的版本号设置的是2.7.0，如果说koa第二位的版本号升级至2.15.0，那么项目执行npm install时，会下载的版本就是2.15.0，换句话说，下载kao的最低版本不会低于2.7.0
  • ~ 代表的是三位版本号中的第三位向上兼容，同样拿koa做例子，当前项目中koa的版本号是2.7.0，如果说koa第三位的版本号升级至2.7.5，那么项目执行npm install时，会下载的版本就是2.7.5
  • 如果不加符号，执行npm install时，会拉取的该大版本下面最新的版本2.x.x，第二和第三位的最大版本号
devDependencies 开发用依赖包及版本号

  • 开发时才会使用到的依赖包可以存放在这个节点下，譬如单元测试包mocha等

  • 安装命令也有些许不同

那package.lock.json又是干什么的呢？

  • 顾名思义，这个文件是用来锁定依赖版本的，是在 npm install时候生成一份文件，用以记录当前状态下实际安装的各个npm package的具体来源和版本号.
  • 尤其在多人同时开发时，可以保证大家所下载的依赖包版本是一致的，避免同一个大版本不同版本号之间存在差异，导致依赖库包行为特征有时候不兼容。
